# Whitelabel

The Whitelabel model is designed for partners who require complete ownership of the user experience and payment orchestration. In this "headless" infrastructure, users are not required to create a Monerium account or interact with Monerium-branded interfaces. Instead, the partner utilizes the API to manage the entire lifecycle — from signup and KYC submission to IBAN issuance and payment execution.

A defining characteristic of the Whitelabel flow is user isolation. Because the partner handles onboarding and "owns" the customer relationship, these users are exclusive to that specific application. Unlike the OAuth ecosystem, where user profiles and balances can be shared or moved between different apps, Whitelabel profiles are strictly siloed. No other Monerium applications can access, view, or interact with these users' data or accounts.

#### Who is it for?

- **Established platforms:** Ideal for partners who already perform their own KYC/KYB and want to avoid redundant compliance checks for their users.
- **Brand-first businesses:** Neobanks, card issuers, or financial services that want to maintain a 100% native brand experience.
- **High-touch entities:** Partners with a few but "large" customers where direct management is preferred.

#### Who is it NOT for?

- **Early-stage startups:** Partners who want to launch quickly without the operational overhead of managing KYC/KYB processes.
- **Community-focused apps:** Projects that want to tap into the existing Monerium user base for "one-click" onboarding.
- **Lightweight antegrations:** Partners looking for the simplest technical path to get started, as Whitelabel requires more complex API implementation and manual privilege approval.

## Authentication

[API Reference →](https://docs.monerium.com/api/#tag/auth)

With **Client Credentials** you can establish a secure connection server-to-server. It allows your backend to obtain the necessary permissions to manage profiles, issue IBANs, and orchestrate payments without requiring a browser-based user login.

### Obtaining credentials

Upon registering your application in either the Sandbox or Production environment, you will be issued a **Client Credentials `client_id`** and a **`client_secret`**.

### Requesting an access token

To authenticate, send a `POST` request to the `/auth/token` endpoint using the following parameters:

- **`grant_type`:** Must be set to `client_credentials`.
- **`client_id`:** Your unique application identifier.
- **`client_secret`:** Your private application secret.

### Managing tokens

A successful authentication request returns two tokens:

- **Access token:** Must be included in the `Authorization: Bearer` header for all subsequent API calls. Valid for **one hour**, after which a new token must be requested.
- **Refresh token:** Used to renew your session without re-submitting your client secret.

### Best practices for token storage

- **Refresh tokens:** Store these exclusively in secure, server-side storage (such as an encrypted database or a dedicated secrets manager). They should never be exposed to the frontend or stored in localStorage.
- **Access tokens:** These are short-lived (valid for one hour). Treat them as session keys and keep them in-memory where possible.
- **Encryption at Rest:** Ensure any database containing these tokens is encrypted to prevent data leaks.

### Key implementation details

- **Authentication context:** In version 2 of the API, the identity of the caller is derived from the token context. By default, requests return data for the application's own profile.
- **Accessing user data:** To perform actions for a specific user (such as fetching a customer's balance or placing a payment), you must explicitly include the user's **profile UUID** in the request arguments.

## Webhooks

[API Reference →](https://docs.monerium.com/api/#tag/webhooks)

For **Whitelabel** integrations, webhooks are the primary mechanism for managing the asynchronous "headless" nature of the API. Since your application handles many siloed users, webhooks allow your backend to react in real-time to compliance approvals and payment events without constant polling.

### Setting up webhooks

To begin receiving notifications, you must create a subscription using the Webhooks endpoint.

- **Registration:** Use your Client Credentials access token to register your target URL.
- **Verification:** Monerium will send a `subscription.created` event to your URL. Your server must respond with a `200 OK` status to activate the subscription.
- **Secret key:** Upon creation, you will receive a webhook secret. Store this securely, as it is required to verify the authenticity of incoming messages.

### Security and signature verification

To ensure a notification was sent by Monerium, every request includes a `webhook-signature` header.

- **Signature method:** The signature is an HMAC-SHA256 hash of a formatted envelope containing the `webhook-id`, `webhook-timestamp`, and the raw request body.
- **Validation:** Your server should compute the signature using your secret and compare it to the header before processing the data.

### Core event types

#### Profile updates (`profile.updated`)

This event is the engine of your automated onboarding flow. It is triggered whenever there is a change in the top-level `ProfileState` or one of the granular `SectionState` attributes (`details`, `form`, or `verifications`).

- **Logic for `approved`:** When the profile state reaches `"approved"`, your backend can trigger the provisioning of a dedicated IBAN.
- **Logic for `incomplete`:** If a profile transitions to or remains in `"incomplete"`, your system must inspect the `requirements` array. Use this list to notify your user exactly which data, such as a specific form or a missing verification - is required to move forward.
- **Logic for `submitted`:** The profile is now under manual review. During this stage, your UI should prevent further data edits.

#### IBAN readiness (`iban.updated`)

Once a dedicated IBAN is requested for a wallet, this event confirms when it is fully provisioned and ready to receive funds.

#### Payment progress (`order.created` and `order.updated`)

- **Incoming (issue) orders:** Notifies you automatically when a user receives a SEPA transfer to their IBAN and EURe is minted to their wallet.
- **Outgoing (redeem) orders:** Tracks the lifecycle of payments sent from the wallet to a bank account, confirming when the transfer is successfully processed.

### Best practices for implementation

- **Respond promptly:** Always return a `200 OK` response immediately upon receiving the webhook. Perform heavy processing or database updates asynchronously to avoid timeouts.
- **Retry logic:** If your server is down, Monerium will retry delivery with exponential backoff up to 10 times over a 12-hour window.
- **Idempotency:** Use the `webhook-id` to ensure your system does not process the same event multiple times if a retry occurs.

## Create profile

[API Reference →](https://docs.monerium.com/api/#tag/profiles/operation/create-profile)

Creating a profile is the foundational step in a Whitelabel integration. This process establishes a "headless" identity within the Monerium ecosystem that is managed entirely through your application. This profile acts as the permanent anchor for all subsequent lifecycle actions, such as KYC/KYB submission, wallet linking, and the issuance of dedicated IBANs.

### The API request

To register a new customer in your ecosystem, send a `POST` request to the `/profiles` endpoint using your **Client Credentials** access token.

- **Endpoint:** `POST /profiles`
- **Header:** `Accept: application/vnd.monerium.api-v2+json`
- **Identity generation:** If you do not supply a specific identifier in the request, the Monerium API will automatically generate a unique **Profile UUID** for the user.
- **Body parameters:** The request requires you to define the nature of the entity being created.

```json
{
  "kind": "personal"
}
```

Use `"personal"` for individuals or `"corporate"` for businesses.

### Handling the response

- **Success (201 Created):** The API returns the full **Profile object**.
- **Data storage:** You must capture and store the `id` (the Profile UUID) in your database. This unique identifier is mandatory for every subsequent API call related to this specific customer.

### User privacy and isolation

Because you are utilizing the Whitelabel application type, the profiles you create are **exclusive to your application**.

- **Siloed profiles:** These profiles are strictly siloed and are not visible to other Monerium partners.
- **Closed-loop ecosystem:** Unlike the OAuth model, where users can share balances between different apps, Whitelabel users stay entirely within your native ecosystem.

## Onboarding (KYC sharing vs. reliance)

[API Reference →](https://docs.monerium.com/api/#tag/profiles/operation/profile)

The onboarding phase is the critical process of supplying regulatory data to move a profile from its initial state toward approval. Monerium provides two distinct technical strategies — **KYC Sharing** and **KYC Reliance** — to streamline this verification while maintaining strict compliance.

### Strategy 1: KYC sharing (automated SumSub)

This path is optimized for partners who use **SumSub** for their primary identity verification. It allows you to programmatically "share" a user's existing verification status with Monerium's compliance engine.

#### 1. Initiate sharing

- **Endpoint:** `POST /profiles/{profileId}/share`
- **Mechanism:** Initiates the transfer of verification data from your SumSub account to Monerium. You must provide the unique SumSub applicant ID associated with the user.

#### 2. Submit regulatory form (required)

- **Endpoint:** `PUT /profiles/{profileId}/form`
- **Purpose:** Even with shared data, this step is **mandatory** for all Sharing integrations.
- **Content:** This form collects specific e-money disclosures (such as "Expected monthly volume") that are required by Monerium's internal policies but are typically not included in a standard SumSub report.

#### 3. Additional verification

- **Endpoint:** `PUT /profiles/{profileId}/verifications`
- **Trigger:** If the onboarding process concludes that the shared SumSub data is insufficient, an additional verification requirement will be triggered. You will need to use this endpoint to provide any specifically requested proofs.

### Strategy 2: KYC reliance (manual submission)

KYC Reliance is designed for licensed partners who manage their own internal compliance and "push" validated customer details directly to Monerium.

#### 1. Submit identity details

- **Endpoint:** `PUT /profiles/{profileId}/details`
- **Requirement:** Submit the foundational KYC/KYB information, such as full name, date of birth, and residency details.

#### 2. Submit KYC questionnaire

- **Endpoint:** `PUT /profiles/{profileId}/form`
- **Requirement:** Submit supplementary data required for specialized account types or to fulfill specific regulatory "Know Your Customer" questions.

#### 3. Upload supporting documents

- **Endpoint:** `POST /files`
- **Process:** You must first upload the raw verification proofs (e.g., ID photos) to obtain a `fileId`, which is then used in the final step.

#### 4. Submit verifications & source of funds

- **Endpoint:** `PUT /profiles/{profileId}/verifications`
- **Process:** This is the final step where you link the customer details to their actual verification proofs using the `fileId`.
- **Source of Funds (SoF):** If the onboarding decision determines that a Source of Funds verification is required for the user (e.g., due to high volume or risk profile), you must include the relevant SoF document `fileId` (like a bank statement or pay stub) within this submission.

### Onboarding status monitoring

Every attribute submitted during onboarding — including the forms, details, and individual documents — has its own independent status. Refer to the next section, **Profile Approval and States**, to understand how to interpret whether a specific form has been accepted or if a document needs resubmission.

## Profile approval and states

[API Reference →](https://docs.monerium.com/api/#tag/profiles/operation/profile)

The Monerium profile uses a multi-layered state machine to manage compliance workflows. To build a robust Whitelabel integration, you must track both the **Profile State** (overall lifecycle) and the **Section States** (the status of specific data groups like details, forms, and verifications).

### Top-level profile states

The `state` attribute on the profile object indicates the user's current stage in the onboarding and operational lifecycle.

| State          | Description                                                                                                                                                    |
| :------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **created**    | The profile has been initialized, but no data has been submitted yet.                                                                                          |
| **incomplete** | One or more sections require additional information. In this state, the partner is permitted to submit or update data.                                         |
| **submitted**  | All required information has been provided, and the profile is awaiting manual review. Submissions are blocked until the state transitions out of `submitted`. |
| **approved**   | The profile is active. All Monerium services, including IBAN issuance and payments, are available.                                                             |
| **rejected**   | Final rejection. The applicant details did not meet compliance requirements, and the lifecycle is terminated.                                                  |

### Section-level states

The individual components of a profile — specifically **`details`**, **`form`**, and **`verifications`** — each maintain their own state. This allows your application to provide granular feedback on which part of the onboarding process is complete or needs attention.

| State          | Description                                                                                                    |
| :------------- | :------------------------------------------------------------------------------------------------------------- |
| **incomplete** | The section requires more information. You can still submit or update data for this specific section.          |
| **submitted**  | All required information for this section has been provided and is awaiting review.                            |
| **approved**   | The section has been successfully reviewed and approved by compliance.                                         |
| **rejected**   | Final rejection for this specific section. No further resubmissions are possible for this part of the profile. |

### The requirements array and transitions

The transition between states is governed by the **`requirements`** array.

- **To move from `created` to `incomplete`**: Submit any initial data (e.g., identity details).
- **To move from `incomplete` to `submitted`**: You must fulfill every item listed in the `requirements` array. Once the array is empty, the profile automatically transitions to `submitted`.
- **Review Loop**: If the compliance team requires a correction, they may transition the profile state from `submitted` back to `incomplete` and add the specific missing item (e.g., `source-of-funds`) back into the `requirements` array.

### Monitoring via webhooks

Because these transitions are asynchronous, your backend must react to the **`profile.updated`** webhook rather than using polling.

- **Success Logic**: When you receive a webhook where the top-level `state` is `"approved"`, your system can proceed to link a wallet and issue an IBAN.
- **Intervention Logic**: If a section moves to `"rejected"`, the process stops. If a section is `"incomplete"`, your UI should prompt the user to provide the specific data requested in the `requirements` array.

:::tip
To ensure a high approval rate, align your compliance team with Monerium's standards early in the integration process to prevent unnecessary rejections.
:::

## Link wallet

[API Reference →](https://docs.monerium.com/api/#tag/addresses)

To enable financial operations, you must associate a specific wallet address with the user's Monerium profile. This connection serves as the permanent bridge between the legacy banking world and the decentralized ledger. Linking a wallet is a mandatory prerequisite for issuing dedicated IBANs and orchestrating automated payments.

### Supported wallet types

Monerium is chain-agnostic and compatible with any wallet capable of providing digital signatures. These generally fall into two categories:

- **Externally Owned Accounts (EOAs):** Standard wallets (like MetaMask or Ledger) controlled directly by a private key.
- **Smart Contract Wallets:** Advanced, programmable accounts (like Safe or Argent). Because these wallets don't have a private key, they utilize the **EIP-1271** standard to authorize actions via code logic.

### On-chain vs. off-chain

The choice between these two layers is about where the "proof" of an action is stored and who pays for it.

| Feature          | Off-chain (Private Proof)                             | On-chain (Public Record)                                |
| :--------------- | :---------------------------------------------------- | :------------------------------------------------------ |
| **Data Storage** | Proof is sent to Monerium's private database via API. | Proof is written directly into the blockchain ledger.   |
| **Cost**         | **Free.** Signing a message costs no "gas."           | **Costs Gas.** You pay the network to record the event. |
| **Speed**        | **Instant.** No waiting for network confirmation.     | **Variable.** Dependent on block times.                 |
| **Visibility**   | **Private**. Only you and Monerium see this link.     | **Public**. Visible to anyone on a block explorer.      |

### Distinguishing between linking and transacting

It is vital to distinguish when these layers are used:

1. **Linking a Wallet:** This is almost always an **off-chain** proof of ownership. You are simply telling Monerium's database, "I own this address." No money moves, and the blockchain is not updated.
2. **Signing a Transaction (Orders):** When you move funds (e.g., burning EURe to send a SEPA payment), the signature you provide is used to trigger an **on-chain** financial event. This changes the state of the blockchain and requires gas.

### The Safe wallet

A Safe wallet is unique because it is controlled by **Owners** rather than a single key. This leads to two specific paths for linking:

#### The off-chain path (EIP-1271)

This is the standard, recommended method. Instead of the Safe "signing" something, the individual Owners sign a message. Monerium then uses the **EIP-1271** standard to "call" the Safe contract and ask: _"Your owners just sent me these signatures. Based on your internal rules (like a 2-out-of-3 requirement), do you agree they are allowed to link this wallet?"_

- **Why use it?** It is 100% secure, free, and provides an instant "Web2-style" experience without touching the blockchain.

#### The on-chain path

You can also link a Safe by sending a transaction directly to the blockchain (using the `signMessage` function). This records an "Authorized" flag inside the Safe's code on the public ledger.

- **Why use it?** Use this only if your users (like large corporations) require an immutable, public audit trail of every permission granted.

### The ownership verification handshake

To ensure security and prevent unauthorized linking, Monerium employs a three-step cryptographic handshake:

1. **The Challenge:** The Monerium API provides a unique, randomized message string.
2. **The Signature:** The user signs this message with their wallet. This is an off-chain action that proves they possess the authority to control that address.
3. **The Verification:** Your application submits this signature to Monerium. The API verifies the signature—either against a private key (EOA) or a contract (Smart Wallet)—and permanently links the address to the user's profile.

## IBAN

[API Reference →](https://docs.monerium.com/api/#tag/ibans)

Once a profile is approved and a wallet address has been successfully linked, you can provision a dedicated IBAN for that user. This IBAN serves as the primary gateway between the traditional Euro banking system (SEPA) and the blockchain.

### Prerequisites for IBAN issuance

- **Approved profile:** The user's KYC/KYB status must be "approved".
- **Linked wallet:** A blockchain address must be successfully associated with the profile via a verified signature.

### The provisioning process

Requesting an IBAN is an asynchronous process managed through the API:

- **Request:** Call the IBAN creation endpoint specifying the profile and the linked wallet address.
- **Timing:** Provisioning typically takes only a few seconds.
- **Confirmation:** Your system will be notified via the `iban.updated` webhook once the IBAN is active and ready for use.

### Moving an IBAN

If a user changes their preferred wallet or moves to a different blockchain, your application can programmatically move their existing IBAN.

- **Mechanism:** Use the `PATCH /ibans` endpoint to re-associate an existing IBAN with a different linked wallet address.
- **Note:** For security and compliance, moving an IBAN is a sensitive action that should be clearly communicated to the user within your interface.

### Capabilities of the Monerium IBAN

- **1st and 3rd party payments:** The IBAN can send and receive payments both to the user's own accounts and to third-party bank accounts.
- **No amount limits:** There are no inherent limits on the transaction amounts these IBANs can handle.
- **Automated logic:** Any funds received at this IBAN are automatically minted as EURe to the connected blockchain wallet within approximately 5 seconds.
- **Gasless minting:** Monerium handles all blockchain gas costs associated with the automated minting of tokens to the user's wallet.

## Orders

[API Reference →](https://docs.monerium.com/api/#tag/orders)

In the Monerium ecosystem, **Orders** are the primary objects used to move funds between the traditional banking system (SEPA) and the blockchain, or between different blockchain networks. For a Whitelabel integration, your backend manages these orders using Client Credentials to automate treasury and user payment flows.

### SEPA payments

SEPA payments bridge the traditional banking network with your users' blockchain wallets. These follow two distinct paths based on the direction of funds.

#### Incoming SEPA payments (issue orders)

- **Automation:** These orders are **fully autonomous**. You do not need to place an order via the API; Monerium creates it automatically upon detecting incoming funds.
- **Mechanism:** When a user receives a SEPA transfer to their dedicated IBAN, the system automatically mints the equivalent EURe tokens to the connected wallet.
- **Counterpart:** Automated based on the incoming bank transfer.
- **Monitoring:** Your backend should listen for the `order.created` webhook to detect when a new autonomous minting process begins.

#### Outgoing SEPA payments (redeem orders)

- **Manual action:** These payments must be **explicitly triggered** via the API.
- **Mechanism:** Monerium "burns" the equivalent EURe tokens from the linked wallet and sends a standard SEPA credit transfer from the user's dedicated IBAN.
- **Counterpart:** Defined using an IBAN identifier within the `counterpart` object.
- **Execution time:** Typically processes within **5 seconds**.

### Bridging / Cross-chain

Bridging allows for the seamless movement of EURe tokens between different supported blockchain networks (e.g., from Ethereum to Gnosis).

- **Mechanism:** A bridge transaction is **always triggered** by the partner. It initiates a **redeem order** on the source chain (to burn tokens). Subsequently, a dependent **issue order** is created on the destination chain (to mint tokens).
- **Logic:** The source redeem order is the "master" order; it does not finalize unless the destination issue order is successfully processed, ensuring funds are never lost in transit.
- **Counterpart:** Defined by a blockchain address and a chain identifier (e.g., `ethereum`, `gnosis`).
- **Fees:** Monerium handles all gas costs for these transactions, and there are no service fees for bridging.

### General redeem order requirements

- Every redeem order requires a cryptographic signature from the linked wallet to authorize the movement of tokens.
- For any redeem order exceeding **15,000 EUR**, you must include a `supportingDocumentId` (e.g., an invoice or contract). This document must be uploaded via the `/files` endpoint before placing the order.

### Order webhooks

Webhooks are essential for tracking the asynchronous lifecycle of both autonomous and triggered orders.

**Events:**

- **`order.created`:** Triggered when a new order is initialized (automatically for incoming SEPA or via API for outgoing/bridge).
- **`order.updated`:** Triggered when an order changes state (e.g., from `placed` to `processed` or `rejected`).

**Security:** All notifications include a `webhook-signature` header. Your server must verify this HMAC-SHA256 signature using your shared secret to ensure the payload is authentic.

### Order endpoints

- **`POST /orders`**: Used to initiate a new redeem order (SEPA payout or cross-chain bridge).
- **`GET /orders`**: Retrieves a list of all orders associated with a specific profile. Can be filtered by `kind`, `state`, or `chain`.
- **`GET /orders/{orderId}`**: Fetches the detailed status and metadata for a specific transaction, including on-chain `txHashes`.

## On-chain transfers and Monerium tokens

The final phase of an integration involves interacting directly with Monerium's suite of emoney tokens on-chain. Since Monerium issues regulated e-money as native blockchain tokens (such as **EURe** and **GBPe**), you can automate sophisticated financial flows — including decentralized exchanges, automated payroll, or treasury rebalancing — directly from your application's backend.

### Monerium smart contracts

Monerium tokens are designed as programmable, 1:1 backed emoney. The smart contract infrastructure is consistent across all issued currencies, ensuring a uniform developer experience.

- **Open source & audited:** All Monerium smart contracts are public and have undergone rigorous professional security audits.
- **Repository:** [Monerium Smart Contracts GitHub](https://github.com/monerium/smart-contracts)
- **Third-party integration:** Monerium tokens are compatible with major DeFi protocols (e.g., CowSwap or Uniswap), allowing users to swap between different Monerium currencies or other digital assets at market prices.

### Token addresses

Contract addresses for all supported chains are listed on the [Tokens](./tokens) page.

### Implementation standards

#### ERC-20 (standard token)

The core of EURe follows the standard **ERC-20** interface, ensuring out-of-the-box compatibility with almost every wallet, exchange, and DeFi protocol in the ecosystem.

- [EIP-20 Specification →](https://eips.ethereum.org/EIPS/eip-20)

#### ERC-2612 (Permit)

This standard allows for "gasless" approvals. Instead of a user having to send a separate transaction to approve a spender, they can sign a message off-chain.

- **Benefit:** Your application can take the signature and submit it with the transfer in a single transaction, significantly reducing friction for the end user.
- [EIP-2612 Tutorial →](https://github.com/monerium/smart-contracts/blob/main/docs/permit.md)
- [EIP-2612 Specification →](https://eips.ethereum.org/EIPS/eip-2612)

## Going live

The Whitelabel application requires thorough preparation due to its "headless" nature and the high level of control it provides over user profiles.

1. **Verified Profile:** The profile owner must complete the full KYC (for individuals) or KYB (for businesses) process and reach an **Approved** state.
2. **Sandbox Validation:** You must create your application in the **Sandbox** environment. It is essential to test your onboarding logic, webhook handlers, and payment flows thoroughly using test blockchains and simulated funds.
3. **Request Production Access:** After successful sandbox testing, create the application in **Production**. You then submit a request for production access, which involves a manual review of your integration to ensure it meets Monerium's standards.
